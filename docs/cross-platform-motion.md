# 크로스플랫폼 모션 가이드

작성일: 2026-07-24

## 목표

동일한 웹 UI를 브라우저, iOS WKWebView, Android WebView에서 실행할 때 화면 진입 애니메이션의 시작 시점과 체감 시간을 일관되게 유지한다.

## 화면 진입 애니메이션 시작

컴포넌트가 DOM에 마운트된 시점과 사용자가 초기 프레임을 실제로 보는 시점은 일치하지 않을 수 있다. 마운트와 동시에 CSS 애니메이션을 실행하면 플랫폼별 초기 렌더링 시간 동안 `animation-delay`가 먼저 소모되어 순차 진입 효과의 앞부분이 보이지 않을 수 있다.

마운트 직후 자동으로 실행되는 진입 애니메이션은 `useDeferredAnimationStart`로 초기 상태와 애니메이션 활성화를 분리한다.

```tsx
const animationReady = useDeferredAnimationStart();

return (
  <Frame data-animation-ready={animationReady}>
    <Content />
  </Frame>
);
```

```css
opacity: 0;
transform: translateY(14px);

[data-animation-ready="true"] & {
  animation: reveal 620ms cubic-bezier(0.22, 1, 0.36, 1) both;
}
```

훅은 두 번의 `requestAnimationFrame`을 기다린 후 `true`를 반환한다.

1. 첫 프레임에서 초기 DOM과 스타일을 렌더링 파이프라인에 등록한다.
2. 다음 프레임에서 초기 상태와 애니메이션 상태가 하나로 합쳐지지 않도록 분리한다.
3. 준비 상태가 된 뒤 모든 순차 애니메이션의 타임라인을 시작한다.

이 방식은 물리적인 디스플레이의 페인트 완료 이벤트를 감지하는 기능이 아니다. 브라우저가 초기 상태를 처리할 프레임을 확보한 뒤 다음 상태를 적용하는 렌더링 스케줄링 규칙이다.

## 적용 대상

- 새 화면이 마운트되면서 자동으로 시작하는 진입 애니메이션
- 여러 요소가 `animation-delay`를 두고 순차적으로 등장하는 화면
- 로고, 이미지 또는 복잡한 배경을 포함한 인트로와 완료 화면

## 적용하지 않는 대상

- 버튼 눌림이나 선택 상태처럼 사용자 입력에 즉시 반응해야 하는 효과
- 드래그, 스와이프처럼 포인터 위치를 연속적으로 따라가는 인터랙션
- 이미 표시된 요소의 단순 상태 전환
- 라우트 전환 진행도와 직접 연결된 인터랙티브 애니메이션

연속 제스처는 별도로 포인터 좌표를 저장하고 `requestAnimationFrame`마다 `transform`을 갱신한다. 진입 애니메이션 훅으로 입력 반응을 지연시키지 않는다.

## 접근성

`prefers-reduced-motion: reduce`에서는 전역 모션 정책을 우선한다. 훅은 동일하게 준비 상태를 만들지만 CSS에서 애니메이션 시간을 제거하거나 짧은 페이드로 대체한다.

## 현재 적용 화면

- `/entry`의 추천 준비 완료 화면

이 화면은 변경 전 iOS에서 순차 지연 구간이 첫 표시 전에 소모되어 약 0.6초만 보였다. 훅 적용 후 iOS와 Android 모두 시각적 완료 시간이 약 0.8초로 일치했다.
